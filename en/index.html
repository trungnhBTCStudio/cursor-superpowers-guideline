🏠 Overview

What is Superpowers?

A collection of skills, hooks, agents and commands that turn Cursor IDE into a real AI developer with proper workflow — instead of just smart autocomplete.

ℹ️
Origin: Superpowers was created by Jesse Vincent (obra), with over ... GitHub stars. Distributed via Cursor Marketplace and installable with /add-plugin superpowers in Agent mode.
4 Core Components of Superpowers
  • ⚡
    Skills (14 skills)Rules in .cursor/rules/*.mdc — tells AI how to handle each task. Invoke with @skill-name or auto-triggers.
  • 🪝
    Hooks (1 hook: SessionStart)Runs automatically when starting a new session — injects context, checks project status.
  • 🤖
    Agents (1 agent: code-reviewer)Specialized subagent that automatically reviews code after each important step completion.
  • 🛠️
    Commands (3 commands)Slash commands /brainstorm, /write-plan, /execute-plan — shortcuts for the 3 most common workflows.

⚙️ Installation

Install Superpowers

Two methods, each with Global (all projects) or Per-project (single project). Source code: github.com/obra/superpowers

ℹ️
Global vs Per-project:
• 🌐 Global: Install once, applies to all projects you open in Cursor. Files stored in ~/.cursor/
• 📁 Per-project: Only applies to 1 project specifically. Files stored in <project>/.cursor/. Ideal when you want per-project config or share via Git.
Method 1 — Cursor Marketplace (Recommended) · 🌐 Global or 📁 Per-project
💡
When installing from Marketplace, Cursor will ask you to choose:
• 🌐 Add to Account: Install for your logged-in account → applies to all projects
• 📁 Add to Project: Only install for the currently open project → applies to 1 project specifically
Cursor Agent Chat
# Method A: Open Cursor → Ctrl+L (or Cmd+L) → select "Agent" mode → type: /add-plugin superpowers # → Choose "Add to Account" (🌐 Global) or "Add to Project" (📁 Per-project) # Method B: Search in Marketplace # Open Cursor → search "superpowers" in plugin marketplace # → Click Install → choose Account or Project # Method C: Install from web browser https://cursor.com/marketplace/superpowers # → Log in → Click "Install"
✅
Verify after install: Close and reopen Cursor. Open Agent chat, type / — you should see 2 groups:
• Skills: /brainstorming, /writing-plans, /executing-plans,… (14 skills)
• Commands: /brainstorm, /write-plan, /execute-plan (3 commands)

• Easy Updates: Just git pull in the skills directory.

🗺️ Basic Workflow

Standard Development Workflow

Superpowers defines a clear development loop from start to finish. Each step uses the right skill, agent, or command.

💡
At each session start: Hook SessionStart auto-runs — injects project context and checks status before your first command.
Complete loop from idea → merged PR
1
💡 Brainstorm — Refine your idea
Describe feature → AI asks Socratic questions → approve design doc
@brainstorming · or /brainstorming
2
🌿 Git Worktree — Create isolated workspace
New branch + isolated worktree + verify clean test baseline
@using-git-worktrees or /using-git-worktrees
3
📋 Write Plan — Detailed blueprint
Decompose design → 2–5 min tasks with exact paths + tests
@writing-plans · or /writing-plans
4
⚡ Execute — Run plan (pick 1 of 2)
With control: @executing-plans — review each batch · Automatic: @subagent-driven-development — 2-stage review
@executing-plans · or /executing-plans · or @subagent-driven-development
5
🧪 TDD — Throughout implementation
RED (test fail) → GREEN (minimal code) → REFACTOR — auto-triggers
@test-driven-development · auto-trigger
6
👀 Code Review — After each task completion
Dispatch code-reviewer agent → receive feedback → handle structured
/requesting-code-review or @requesting-code-review → agent: code-reviewer → @receiving-code-review
7
✅ Verification — Before claiming "done"
Check all requirements, tests, security — no skipping
@verification-before-completion · auto-trigger
8
🏁 Finish Branch — Merge / PR / Discard
Pre-flight check → choose option → clean up worktree
@finishing-a-development-branch or /finishing-a-development-branch
🔀
Parallel mode: If the plan has independent tasks, add a step after #3: use @dispatching-parallel-agents to run multiple tasks simultaneously — 3–5x speedup.
🪝 Hooks

Hooks — Auto-run at session start

Hooks are code that runs automatically on trigger events in Cursor — no manual invocation needed. Superpowers currently has 1 hook: SessionStart.

ℹ️
Hooks differ from Skills: Skills need you to invoke (@skill-name) or auto-trigger by pattern. Hooks run automatically on system events — like middleware in a web framework.
SessionStartAuto-trigger

🪝 SessionStart Hook

Runs immediately when you open a new Cursor Agent session. This hook injects project context at the start of the conversation — AI knows your project before you ask anything.

Defined in
hooks/hooks-cursor.json
{ "hooks": [ { "name": "SessionStart", "trigger": "onSessionStart", "description": "Hook: SessionStart", "action": "injectProjectContext" } ] }
What does SessionStart Hook do?
  • 1
    Read project contextFind README, package.json, .cursorrules to understand the project
  • 2
    Inject into conversationAI "remembers" tech stack, conventions and coding standards from the start
  • 3
    Verify available skillsReports which skills are currently active in .cursor/rules/
💡
Optimize hook: Add a .cursorrules or docs/project-context.md file to the project root — SessionStart will automatically inject it at the start of every conversation.
🤖 Agents

Agents — Specialized Subagents

Agents are pre-defined AI subagents with specific roles. Superpowers has 1 agent: code-reviewer — automatically invoked when an important step is completed.

ℹ️
Agent vs Skill: A Skill guides the current AI on what to do. An Agent spawns a new independent AI with a specialized role — this reviewer is not biased by the implementer's conversation history.
code-reviewerAuto-triggered

🤖 Agent: code-reviewer

This agent is automatically invoked when a major step in the project is completed. It reviews written code against the original plan and coding standards — not a rubber-stamp, but a genuine review.

When is this agent invoked?
  • ▶
    After completing a major step"I've finished implementing the user authentication system as outlined in step 3 of our plan"
  • ▶
    After completing a feature implementation"The API endpoints for the task management system are now complete — that covers step 2 from our architecture document"
Sample prompt to trigger code-reviewer agent
Use together with @requesting-code-review
@requesting-code-review I've finished implementing the auth feature — Task 3 from docs/superpowers/plans/YYYY-MM-DD-<feature-name>.md WHAT_WAS_IMPLEMENTED: Login controller with JWT auth and bcrypt + rate limiting GIT_RANGE: BASE_SHA: abc123 (before Task 3) HEAD_SHA: HEAD Please dispatch code-reviewer agent to review against: - Original plan requirements - Coding standards in .cursorrules - Security best practices
💡
Why use a separate agent? The reviewer agent has a clean context window — not influenced by the implementer's "I just wrote this code" history. Significantly more objective review.
Sample output from code-reviewer agent
code-reviewer agent output
Reviewing: Task 3 — Login Controller Against: docs/superpowers/plans/YYYY-MM-DD-<feature-name>.md + .cursorrules ✓ Strengths: - Clean separation: controller → service → utils - Real tests (not just mocks) — 8 tests covering all edge cases - bcrypt timing-safe comparison implemented correctly ⚠ Issues Found: 🔴 CRITICAL [auth/login.controller.js:47] SQL string interpolation — use parameterized queries Fix: db.query('SELECT * FROM users WHERE email = $1', [email]) 🟡 IMPORTANT [auth/login.controller.js:89] Missing Retry-After header on 429 response Add: res.set('Retry-After', 900) ⚪ MINOR: Magic number 5 (max attempts) — move to config Assessment: BLOCKED — fix CRITICAL before proceeding

Skill 1 — Core

💡 Brainstorming — Refine ideas into specs

The first and mandatory skill. Activates before any creative work. AI asks Socratic questions in chunks to clarify requirements and create a design document.

💡
@brainstorming · brainstorming.mdc · /brainstorm
Explores user intent, requirements and design before implementation. MUST USE before any creative work.
Auto-trigger Saves design doc Socratic questions
⚠️
Auto-trigger: This skill activates as soon as you describe a feature you want to build. Just say "I want to build X" and Superpowers starts asking questions — no manual invocation needed.
Sample prompt
Start brainstorm with spec
@brainstorming I have a spec for [System Name]. [Attach file or paste spec content] Goal: Refine this into a production-ready design that is clear, unambiguous, and can later be implemented by AI. Follow this process strictly: 1. **Explore & Understand:** - Ask questions ONE AT A TIME to clarify: - System goals - Scale (users, traffic, growth expectations) - Constraints (technology, infrastructure, cost, security) - Identify any missing assumptions - If visualization would help (architecture diagrams, UI), suggest using a Visual Companion 2. **Challenge the Design:** - Identify blind spots in the spec - List real-world edge cases (race conditions, retries, partial failures, etc.) - Analyze potential failure scenarios (downtime, data inconsistency, scaling bottlenecks) 3. **Propose Approaches:** - Suggest 2–3 different approaches - For each approach, include: - High-level architecture - Pros - Cons - When it should be chosen 4. **Iterative Design:** - Break the design into small sections: - Architecture - Data model - API design - Data flow - Error handling - For each section, include: - Problem - Proposed solution - Trade-offs - STOP and wait for my confirmation before moving to the next section 5. **Final Output:** - Produce a complete, refined spec in Markdown format - The spec must be: - Clear and unambiguous - Well-structured - Ready for AI-driven implementation **Important:** - DO NOT write code or implementation - DO NOT assume missing details without confirmation - Prioritize robustness, scalability, and real-world reliability over simplicity
Sample output — AI asks questions per chunk
Brainstorm session
== CHUNK 1: Core Authentication == I understand you want email/password login returning JWT. Before I continue, let me clarify: Q1: Should users be able to have multiple active sessions (e.g., logged in on phone + laptop simultaneously)? → Option A: Yes, multiple sessions → Option B: Single session (new login invalidates old token) Q2: Do you need "Remember me" functionality (extended token expiry)? Q3: Is social login (Google/GitHub) in scope now or future? [Waiting for your answers before proceeding to Chunk 2...]
💡
After answering all questions: AI saves the design document to docs/superpowers/specs/YYYY-MM-DD-<topic>-design.md. This file is the input for @writing-plans in the next step.
Skill 2 — Core

🌿 Git Worktrees — Isolated workspace per feature

Activates right after design is approved. Creates an isolated workspace on a new branch, runs project setup, verifies clean test baseline before coding begins.

🌿
@using-git-worktrees · using-git-worktrees.mdc
Creates an isolated workspace on a new branch. Run after design approval, before writing any code.
After design approval Isolated workspace Baseline verify
🆕
Why use git worktrees? Worktrees allow working on multiple features in parallel on the same repo without clobbering each other. Subagents also run in their own worktree to avoid conflicts when using parallel agents.
Prompt — Setup worktree after design approval
Prompt
@using-git-worktrees Design for [Feature Name] has been approved. Please set up an isolated workspace to begin implementation. **Configuration:** - Branch name: [e.g., feature/feature-name] - Base branch: [e.g., main] **Workflow (Strict Skill Compliance):** 1. **Safety Verification:** - Verify if the worktree directory (prefer `.worktrees/`) is properly ignored by git. - If not, add it to `.gitignore` and commit the change before proceeding. 2. **Prepare Base Branch:** - Ensure [Base branch] is up to date (fetch + pull latest). 3. **Worktree Creation:** - Create a new worktree for branch [Branch name] at path `.worktrees/[Dir Name]`. - If the branch or worktree already exists, reuse it instead of creating a new one. 4. **Project Setup & Baseline:** - Navigate into the worktree directory. - Auto-detect and run setup commands (npm install/yarn/etc.). - Run the test suite to verify a clean baseline. 5. **Report Status:** Provide a structured report with: - Full worktree path. - Test results summary (Passing/Failing). - Confirmation: "Ready to implement [Feature Name]". **Important:** Do NOT modify any code or start implementation during this turn. Only prepare and validate the development environment.
Sample output
Worktree setup complete
✓ Created worktree at: ../your-project-auth/ ✓ Branch: feature/user-auth (from main @ a3f821c) ✓ npm install — 847 packages installed ✓ npm test baseline: - 23 tests passing ✓ - 0 failing - 0 skipped Clean baseline confirmed. Ready to implement.
💡
Benefit: If implementation fails, you can discard the worktree without affecting the main branch. Use together with @finishing-a-development-branch at the end.
Skill 3 — Core

📋 Writing Plans — Detailed implementation blueprint

Decomposes design into 2–5 minute tasks. Each task has exact file paths, code to write, and verification steps. Clear enough that "a junior engineer without context can still execute it".

📋
@writing-plans · writing-plans.mdc · /write-plan
Decomposes design into 2–5 minute tasks with exact file paths, complete code, and verification steps.
2-5 min per task Exact file paths Verification per step
Prompt — Create plan from design doc
Prompt
@writing-plans Create a detailed implementation plan from this approved design: [paste content of docs/superpowers/specs/YYYY-MM-DD--design.md] Tech stack: Node.js + Express + PostgreSQL + Jest Goal: Produce a step-by-step, AI-executable plan using strict TDD (RED → GREEN → VERIFY) and checkbox syntax. --- ## Plan Structure Requirements: 1. **Header:** Start the plan document with a standard header (Feature Name, Goal, Architecture, Tech Stack). 2. **Task Granularity:** - Each task MUST be completable in 2–5 minutes. - Each task should modify ONLY 1–2 files max. 3. **Task Format (MANDATORY):** For each task, use the standard Superpowers format: ### Task N: [Component Name] **Files:** - Create/Modify: `path/to/file.js` - Test: `path/to/test.js` - [ ] **Step 1: Write the failing test** (Include code block) - [ ] **Step 2: Run test to verify it fails** (Command + expected FAIL) - [ ] **Step 3: Write minimal implementation** (Include code block) - [ ] **Step 4: Run test to verify it passes** (Command + expected PASS) - [ ] **Step 5: Commit** (Git command with conventional message) 4. **No Placeholders:** - No "TBD", "TODO", or vague logic. - Every step must be fully documented and ready for execution. **Output:** - Save the plan to `docs/superpowers/plans/YYYY-MM-DD-[feature-name].md`. - Ensure it is ready for `subagent-driven-development`.

Skill 4 — Core

▶️ Executing Plans — Execute plan in batches

Run plan in batches with human checkpoints. Use when you want to keep control — review each task group before continuing. Best for small–medium projects.

▶️
@executing-plans · executing-plans.mdc · /execute-plan
Executes implementation plan in batches with human review checkpoints between each batch.
Human checkpoints Batch execution Control > Speed
ℹ️
executing-plans vs subagent-driven-development: Use executing-plans when you want manual review after each batch. Use subagent-driven-development when you want AI to continuously self-review — faster but less control.
Prompt — Execute each batch
Standard Execution Prompt
@executing-plans Execute Task 1 and Task 2 from the implementation plan: docs/superpowers/plans/YYYY-MM-DD-[feature-name].md --- ## Step 1: Critical Review Before taking any coding action: - Read the entire plan document carefully. - Identify any ambiguities, contradictions, or gaps in the TDD steps. - If everything looks correct, announce: "I'm using the executing-plans skill to implement this plan" and wait for my confirmation to begin Task 1. ## Step 2: Execution Rules 1. **Strict Task Execution:** - Execute ONLY the specified tasks. - Do NOT skip, combine, or reorder steps. 2. **TDD Workflow (MANDATORY):** Follow the 3-stage process for each task: - **RED:** Write the failing test first and run it. - **GREEN:** Implement minimal code to pass the test. - **VERIFY:** Re-run all relevant tests to confirm success. 3. **Scope Control:** - ONLY modify files listed in the task. - Do NOT refactor existing code or add features outside the spec. 4. **Monitoring & Checkpoints:** After EACH task, provide a report in this format: - **Task ID:** - **Changes:** (Summary of files created/modified) - **Test Results:** (Command used and pass/fail status) - **Status:** (PASS / FAIL / BLOCKED) - **STOP** and wait for my approval before proceeding to the next task. 5. **Failure Handling:** - If tests fail in the GREEN or VERIFY phase, attempt a minimal fix within the task scope. - If still failing, STOP and report the issue — do NOT proceed to the next task. --- **Important:** Do NOT jump ahead or optimize beyond the current task scope.
Skill 5 — Core

🤖 Subagent-Driven Development — 2-stage auto review

Each task is dispatched to a fresh subagent. After completion, 2 independent reviewers are auto-dispatched: spec compliance and code quality. Extremely fast for large projects.

🤖
@subagent-driven-development · subagent-driven-development.mdc
Each task gets a fresh implementer subagent. Two reviewer subagents auto-review each completed task.
Fresh subagent per task 2-stage auto review Best for large projects
How it works — 4 steps per task
  • 1
    Dispatch implementer subagentSubagent receives task text + context, implements + self-reviews, commits
  • 2
    Dispatch spec-compliance reviewerSeparate reviewer checks: does code match spec? Missing requirements?
  • 3
    Dispatch code-quality reviewerSecond reviewer checks: DRY, error handling, security, performance
  • 4
    Continue or fixIf issues found → implementer fixes. If passed → move to next task
Prompt — Start subagent-driven execution
@subagent-driven-development Execute this implementation plan using subagent-driven development: [read file: docs/superpowers/plans/YYYY-MM-DD-<feature-name>.md] For each task: 1. Dispatch fresh implementer subagent with task text + context 2. After implementation: dispatch spec-compliance reviewer 3. After spec review: dispatch code-quality reviewer 4. Only proceed to next task when both reviewers approve Report status after each task completes.
Status codes — Understanding return values
Subagent status protocol
DONE → Task completed, both reviewers approved ✓ DONE_WITH_CONCERNS → Done but has minor issues — you decide to continue or fix BLOCKED → Subagent needs more context from you NEEDS_CONTEXT → Missing info (API key, config, spec clarification)
Skill 6 — Core

⚡ Dispatching Parallel Agents — Run multiple tasks simultaneously

Dispatch multiple subagents to work on independent tasks in parallel. 3–5x speedup when there are many modules with no dependencies. Requires git worktrees to avoid conflicts.

⚡
@dispatching-parallel-agents · dispatching-parallel-agents.mdc
Use when facing 2+ independent tasks that can be worked on without shared state or sequential dependencies.
3-5x faster Independent tasks only Each in own worktree
⚠️
Mandatory requirement: Tasks must be independent — no dependencies. Example: writing password.js and jwt.js can be parallel. But login.controller.js depends on both → must run after.
Prompt — Dispatch parallel agents
Prompt
@dispatching-parallel-agents These tasks are independent — dispatch them in parallel: Agent 1 — Password utility: Task: Implement src/utils/password.js Tests: src/utils/password.test.js Spec: [paste task 1 from plan] Agent 2 — JWT service: Task: Implement src/utils/jwt.js Tests: src/utils/jwt.test.js Spec: [paste task 2 from plan] Agent 3 — DB schema: Task: Create migrations/001_users.sql Spec: [paste task 3 from plan] Each agent works in its own git worktree. Report when ALL agents complete.
ℹ️
Context isolation: Each subagent only receives the exact context it needs — no shared session history. Prevents context pollution and keeps subagents more focused.
Skill 7 — Core

🧪 Test-Driven Development — RED → GREEN → REFACTOR

You must write a failing test first. Watch it fail. Write minimal code to pass. Watch it pass. Commit. This skill will delete code written before tests exist.

🧪
@test-driven-development · test-driven-development.mdc
Enforces RED → GREEN → REFACTOR. Auto-triggers during implementation. Will delete pre-test code.
Auto-trigger during impl Deletes pre-test code RED GREEN REFACTOR
TDD 3-step flow
Step 1 — RED (write tests first)
@test-driven-development Write FAILING tests for the login controller. Cover from spec: - 200 + JWT on valid creds - 401 on wrong password - 403 on locked account - 429 on rate limit exceeded File: src/auth/__tests__/login.test.js DO NOT write implementation yet. Run tests — confirm they FAIL.
Step 2 — GREEN (minimal code)
Tests are written and FAILING ✓ Now implement MINIMAL code to make all tests pass. File: src/auth/login.controller.js Rules: - Write minimum code to pass - Don't add extra features - Run tests after each change - Stop when all GREEN ✓
Step 3 — REFACTOR (after GREEN)
All tests passing ✓ Now REFACTOR the login controller: - Remove duplication - Improve naming - Extract helper functions if needed Rules: Tests must still pass after every change. Commit after refactor complete.

🐛 Systematic Debugging — 4-phase root cause process

When there's a bug — don't guess, don't try random fixes. 4 phases: Reproduce → Hypothesis → Verify → Fix. Includes root-cause-tracing, defense-in-depth, condition-based-waiting.

🐛
@systematic-debugging · systematic-debugging.mdc
4-phase debugging: Reproduce → Hypothesis → Verify → Fix root cause. Includes 3 specialized techniques.
4-phase process Root cause only Defense-in-depth
Debug template — Copy and fill in
Standard prompt — Systematic debug
@systematic-debugging Bug report: - Expected: [describe expected behavior] - Actual: [describe actual behavior] - When: [conditions that trigger the bug] Error log: [paste full error + stack trace] Relevant files: [file 1], [file 2] Phase 1 (Reproduce): Confirm you can reproduce it Phase 2 (Hypothesis): List 3 possible root causes, ranked Phase 3 (Verify): Test each hypothesis — DO NOT fix yet Phase 4 (Fix): Fix only the confirmed root cause
3 specialized techniques in this skill
  • A
    root-cause-tracingTrace the bug backwards through the call stack from symptom → actual source. Don't fix the symptom, fix the root.
  • B
    defense-in-depthAfter fixing, add validation at multiple layers so this entire class of bugs cannot occur again.
  • C
    condition-based-waitingInstead of setTimeout, poll actual conditions. Avoids race conditions and flaky tests.
