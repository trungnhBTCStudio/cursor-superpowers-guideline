🏠 Tổng quan

Superpowers là gì?

Một bộ skills, hooks, agents và commands giúp Cursor IDE trở thành AI developer thực sự có quy trình — thay vì chỉ là autocomplete thông minh.

ℹ️
Nguồn gốc: Superpowers được tạo bởi Jesse Vincent (obra), có hơn ... GitHub stars. Được phân phối qua Cursor Marketplace và có thể cài bằng /add-plugin superpowers trong Agent mode.

  • ⚡
    Skills (14 skills)Rules trong .cursor/rules/*.mdc — hướng dẫn AI cách làm từng việc. Gọi bằng @skill-name hoặc tự kích hoạt.

⚙️ Cài đặt

Cài đặt Superpowers

Hai cách cài, mỗi cách có tùy chọn Global (tất cả dự án) hoặc Per-project (chỉ 1 dự án). Source code: github.com/obra/superpowers

ℹ️
Global vs Per-project:
• 🌐 Global: Cài một lần, có hiệu lực ở tất cả dự án bạn mở bằng Cursor. File nằm trong ~/.cursor/
• 📁 Per-project: Chỉ có hiệu lực trong 1 dự án cụ thể. File nằm trong <project>/.cursor/. Thích hợp khi muốn mỗi project có config riêng hoặc chia sẻ qua Git.
Cách 1 — Cursor Marketplace (Khuyến nghị) · 🌐 Global hoặc 📁 Per-project
💡
Khi cài từ Marketplace, Cursor sẽ hỏi bạn chọn:
• 🌐 Add to Account: Cài cho tài khoản đang đăng nhập → có hiệu lực ở tất cả dự án
• 📁 Add to Project: Chỉ cài cho dự án đang mở → có hiệu lực ở 1 dự án cụ thể

🗺️ Basic Workflow

Quy trình phát triển chuẩn

Superpowers định nghĩa một vòng lặp phát triển rõ ràng từ đầu đến cuối. Mỗi bước sử dụng đúng skill, agent hoặc command tương ứng.

💡
Bắt đầu mỗi session: Hook SessionStart tự động chạy — inject project context và kiểm tra trạng thái trước khi bạn gõ lệnh đầu tiên.
Vòng lặp đầy đủ từ idea → merged PR
1
💡 Brainstorm — Tinh chỉnh ý tưởng
Mô tả feature → AI hỏi Socratic → approve design doc
@brainstorming · hoặc /brainstorm
2
🌿 Git Worktree — Tạo workspace riêng
Branch mới + worktree riêng + verify test baseline sạch
@using-git-worktrees
3
📋 Write Plan — Blueprint chi tiết
Phân tách design → tasks 2–5 phút với exact paths + tests
@writing-plans · hoặc /write-plan
4
⚡ Execute — Chạy plan (chọn 1 trong 2)
Có control: @executing-plans — review từng batch · Tự động: @subagent-driven-development — 2-tầng review
@executing-plans · /execute-plan · hoặc @subagent-driven-development
5
🧪 TDD — Trong suốt quá trình implement
RED (test fail) → GREEN (minimal code) → REFACTOR — tự kích hoạt
@test-driven-development · auto-trigger
6
👀 Code Review — Sau mỗi task hoàn thành
Dispatch code-reviewer agent → nhận feedback → xử lý có cấu trúc
@requesting-code-review → agent: code-reviewer → @receiving-code-review
7
✅ Verification — Trước khi claim "done"
Check tất cả requirements, tests, security — không skip
@verification-before-completion · auto-trigger
8
🏁 Finish Branch — Merge / PR / Discard
Pre-flight check → chọn option → clean up worktree
@finishing-a-development-branch
🔀
Parallel mode: Nếu plan có các tasks độc lập nhau, thêm bước sau #3: dùng @dispatching-parallel-agents để chạy nhiều task đồng thời — tăng tốc 3–5x.
🪝 Hooks

Hooks — Tự động chạy khi bắt đầu session

Hooks là code chạy tự động theo trigger event trong Cursor — không cần gọi thủ công. Superpowers hiện có 1 hook: SessionStart.

ℹ️
Hooks khác Skills: Skills cần bạn gọi (@skill-name) hoặc auto-trigger theo pattern. Hooks chạy tự động theo event hệ thống — như middleware trong web framework.
SessionStartAuto-trigger

🪝 SessionStart Hook

Chạy ngay khi bạn mở Cursor Agent session mới. Hook này inject project context vào đầu conversation — AI biết project của bạn là gì trước khi bạn hỏi gì.

Định nghĩa trong
hooks/hooks-cursor.json
{ "hooks": [ { "name": "SessionStart", "trigger": "onSessionStart", "description": "Hook: SessionStart", "action": "injectProjectContext" } ] }
Hook SessionStart làm gì?
  • 1
    Đọc project contextTìm README, package.json, .cursorrules để hiểu project
  • 2
    Inject vào conversationAI "nhớ" tech stack, conventions và coding standards ngay từ đầu
  • 3
    Kiểm tra skills có sẵnBáo cáo skills nào đang active trong .cursor/rules/
💡
Tối ưu hook: Thêm file .cursorrules hoặc docs/project-context.md vào root project — SessionStart sẽ tự inject vào đầu mọi conversation.
🤖 Agents

Agents — Subagents chuyên biệt

Agents là AI subagents được định nghĩa sẵn với vai trò cụ thể. Superpowers có 1 agent: code-reviewer — tự động được gọi khi hoàn thành một bước quan trọng.

ℹ️
Agent vs Skill: Skill hướng dẫn AI hiện tại làm gì. Agent spawn một AI mới độc lập với vai trò chuyên biệt — reviewer này không bị bias bởi conversation lịch sử của implementer.
code-reviewerAuto-triggered

🤖 Agent: code-reviewer

Agent này tự động được gọi khi một bước lớn trong project hoàn thành. Nó review code đã viết so với plan gốc và coding standards — không phải rubber-stamp, mà là genuine review.

Khi nào agent này được gọi?
  • ▶
    Sau khi hoàn thành một major step"I've finished implementing the user authentication system as outlined in step 3 of our plan"
  • ▶
    Sau khi hoàn thành một feature implementation"The API endpoints for the task management system are now complete — that covers step 2 from our architecture document"
Prompt mẫu để kích hoạt code-reviewer agent
Dùng kết hợp với @requesting-code-review
@requesting-code-review I've finished implementing the auth feature — Task 3 from docs/plans/auth-plan.md WHAT_WAS_IMPLEMENTED: Login controller with JWT auth and bcrypt + rate limiting GIT_RANGE: BASE_SHA: abc123 (before Task 3) HEAD_SHA: HEAD Please dispatch code-reviewer agent to review against: - Original plan requirements - Coding standards in .cursorrules - Security best practices
💡
Tại sao dùng agent riêng? Reviewer agent có context window sạch — không bị ảnh hưởng bởi lịch sử "tôi vừa mới viết code này" của implementer. Review khách quan hơn đáng kể.
Output mẫu từ code-reviewer agent
code-reviewer agent output
Reviewing: Task 3 — Login Controller Against: docs/plans/auth-plan.md + .cursorrules ✓ Strengths: - Clean separation: controller → service → utils - Real tests (not just mocks) — 8 tests covering all edge cases - bcrypt timing-safe comparison implemented correctly ⚠ Issues Found: 🔴 CRITICAL [auth/login.controller.js:47] SQL string interpolation — use parameterized queries Fix: db.query('SELECT * FROM users WHERE email = $1', [email]) 🟡 IMPORTANT [auth/login.controller.js:89] Missing Retry-After header on 429 response Add: res.set('Retry-After', 900) ⚪ MINOR: Magic number 5 (max attempts) — move to config Assessment: BLOCKED — fix CRITICAL before proceeding

Usage

Usage

Tương đương @executing-plans. Execute plan theo batches với review checkpoints. Dùng khi muốn giữ control — review thủ công sau mỗi batch.

Skill 1 — Core

💡 Brainstorming — Tinh chỉnh ý tưởng thành spec

Skill đầu tiên và bắt buộc. Kích hoạt trước bất kỳ creative work nào. AI đặt câu hỏi Socratic theo từng chunk để làm rõ yêu cầu và tạo design document.

💡
@brainstorming · brainstorming.mdc · /brainstorm
Explores user intent, requirements and design before implementation. MUST USE before any creative work.
Auto-trigger Saves design doc Socratic questions
⚠️
Trigger tự động: Skill này kích hoạt ngay khi bạn mô tả feature muốn build. Chỉ cần nói "Tôi muốn build X" là Superpowers bắt đầu hỏi — không cần gọi thủ công.
Prompt mẫu
Khởi động brainstorm với DOCX
@brainstorming I have a DOCX with requirements for a user authentication system. [attach file / paste nội dung] Help me refine this into a clear design. Ask me questions about: - What I'm NOT sure about - Edge cases I might have missed - Alternative approaches worth considering Present the design in small chunks for me to validate.
Output mẫu — AI hỏi từng chunk
Brainstorm session
== CHUNK 1: Core Authentication == I understand you want email/password login returning JWT. Before I continue, let me clarify: Q1: Should users be able to have multiple active sessions (e.g., logged in on phone + laptop simultaneously)? → Option A: Yes, multiple sessions → Option B: Single session (new login invalidates old token) Q2: Do you need "Remember me" functionality (extended token expiry)? Q3: Is social login (Google/GitHub) in scope now or future? [Waiting for your answers before proceeding to Chunk 2...]
💡
Sau khi trả lời xong: AI lưu design document vào docs/superpowers/specs/YYYY-MM-DD-<topic>-design.md. File này là input cho @writing-plans ở bước tiếp theo.
Skill 2 — Core

🌿 Git Worktrees — Workspace tách biệt cho mỗi feature

Kích hoạt ngay sau khi design được approve. Tạo workspace tách biệt trên branch mới, chạy project setup, verify test baseline sạch trước khi bắt đầu code.

🌿
@using-git-worktrees · using-git-worktrees.mdc
Creates an isolated workspace on a new branch. Run after design approval, before writing any code.
After design approval Isolated workspace Baseline verify
🆕
Tại sao cần git worktrees? Worktrees cho phép làm nhiều feature song song trên cùng 1 repo mà không clobber nhau. Subagents cũng chạy trong worktree riêng để tránh conflict khi dùng parallel agents.
Prompt — Setup worktree sau khi approve design
Prompt
@using-git-worktrees Design for auth feature has been approved. Create an isolated workspace for implementation. Branch name: feature/user-auth Base branch: main After setup: 1. Run npm install (or yarn) 2. Run npm test to verify baseline is clean 3. Report what tests exist and their current status
Output mẫu
Worktree setup complete
✓ Created worktree at: ../your-project-auth/ ✓ Branch: feature/user-auth (from main @ a3f821c) ✓ npm install — 847 packages installed ✓ npm test baseline: - 23 tests passing ✓ - 0 failing - 0 skipped Clean baseline confirmed. Ready to implement.
💡
Lợi ích: Nếu implementation thất bại, bạn discard worktree mà không ảnh hưởng main branch. Dùng kết hợp với @finishing-a-development-branch ở cuối.

Skill 4 — Core

▶️ Executing Plans — Thực thi plan theo batch

Chạy plan theo từng batch có human checkpoints. Dùng khi muốn giữ control — review từng nhóm task trước khi tiếp. Phù hợp với project nhỏ–vừa.

▶️
@executing-plans · executing-plans.mdc · /execute-plan
Executes implementation plan in batches with human review checkpoints between each batch.

Skill 7 — Core

🧪 Test-Driven Development — RED → GREEN → REFACTOR

Bắt buộc viết test thất bại trước. Watch it fail. Viết minimal code để pass. Watch it pass. Commit. Skill này sẽ xóa code được viết trước khi có test.

🧪
@test-driven-development · test-driven-development.mdc
Enforces RED → GREEN → REFACTOR. Auto-triggers during implementation. Will delete pre-test code.
Auto-trigger during impl Deletes pre-test code RED GREEN REFACTOR
Flow TDD 3 bước
Bước 1 — RED (viết test trước)
@test-driven-development Write FAILING tests for the login controller. Cover from spec: - 200 + JWT on valid creds - 401 on wrong password - 403 on locked account - 429 on rate limit exceeded File: src/auth/__tests__/login.test.js DO NOT write implementation yet. Run tests — confirm they FAIL.
Bước 2 — GREEN (minimal code)
Tests are written and FAILING ✓ Now implement MINIMAL code to make all tests pass. File: src/auth/login.controller.js Rules: - Write minimum code to pass - Don't add extra features - Run tests after each change - Stop when all GREEN ✓
Bước 3 — REFACTOR (sau khi GREEN)
All tests passing ✓ Now REFACTOR the login controller: - Remove duplication - Improve naming - Extract helper functions if needed Rules: Tests must still pass after every change. Commit after refactor complete.

🐛
@systematic-debugging · systematic-debugging.mdc
4-phase debugging: Reproduce → Hypothesis → Verify → Fix root cause. Includes 3 specialized techniques.
4-phase process Root cause only Defense-in-depth
Template debug — Copy và điền vào
Prompt chuẩn — Systematic debug
@systematic-debugging Bug report: - Expected: [mô tả behavior mong muốn] - Actual: [mô tả behavior thực tế] - When: [điều kiện trigger bug] Error log: [paste full error + stack trace] Relevant files: [file 1], [file 2] Phase 1 (Reproduce): Confirm you can reproduce it Phase 2 (Hypothesis): List 3 possible root causes, ranked Phase 3 (Verify): Test each hypothesis — DO NOT fix yet Phase 4 (Fix): Fix only the confirmed root cause
3 kỹ thuật đặc biệt trong skill này
  • A
    root-cause-tracingTrace bug ngược qua call stack từ symptom → actual source. Không fix symptom, fix root.

  • C
    condition-based-waitingThay vì setTimeout, poll condition thực tế. Tránh race condition và flaky tests.
